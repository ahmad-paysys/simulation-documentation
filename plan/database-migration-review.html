Executive Summary

The schema is structurally strong for auditability and deterministic replay. Most denormalized elements are intentional snapshots.
  • Current implementation uses only the root table trs_simulation_suites in live TypeScript runtime code.
  • All other tables are design-aligned with the planned run pipeline, but are not yet evidenced in runtime repositories/controllers.
  • The main duplication pattern is snapshot duplication, not accidental duplication: rule identity fields and payload snapshots are repeated to preserve historical fidelity per generation and run.
  • 3NF is mostly preserved at entity level; controlled denormalization exists for performance and immutability.
  • Highest risk is not schema correctness, but delivery drift: large portions of schema are ahead of currently implemented endpoints.

Unused and Duplicate Analysis

Severity High: The migration provisions 17 tables, but runtime code currently references only trs_simulation_suites. This is a delivery gap risk, not a DDL defect.

Potentially Unused Right Now

  • All non-suite tables (16 tables) are currently planned/internal from code perspective, with no direct repository/controller SQL usage evidenced in runtime TypeScript sources.
  • Within trs_simulation_suites, clone_source_suite_id exists but no complete clone endpoint flow is currently evidenced in active runtime code.
  • Within trs_simulation_suites, simulation_type is currently accepted but appears operationally single-mode in implemented paths.

Intentional Duplicates (Snapshot Denormalization)

  • rule_repo and rule_version are repeated in suite, generation, and run tables to freeze historical identity across lifecycle transitions.
  • context_payload_json and trigger_payload_json in trs_suite_context_sim_pairs duplicate payloads already held by message tables, likely for deterministic replay without extra joins.
  • Counter columns in trs_suite_generations (context_count, trigger_count, generated_* and *_field_config_count) are derivable from child rows but likely retained for read efficiency and immutable snapshots.
  • iteration_count, run_count, and last_run_at on trs_simulation_suites are rollup fields derivable from descendants, but useful for list performance.
Recommendation: keep snapshot duplicates that protect determinism/auditability, but classify each duplicate as either "performance cache", "immutability snapshot", or "remove candidate" and enforce owner-level write rules.

rule_repo vs rule_name

They serve different concerns and should not be merged.

Field Meaning Operational Role Observed Usage
rule_repo Technical image/repository identity for executable rule artifact. Execution and reproducibility pipeline (especially with tag/digest on runs). Mapped through create/patch payloads in service and repository; not currently exposed as list filter in implemented endpoints.
rule_name Business-facing display name of selected rule. UI readability, search/filter, operator workflow. Implemented list filter in admin repository (rule_name ILIKE), and surfaced in controller query DTOs.
Practical model: keep both fields, enforce consistency at write boundary, and optionally add a canonical rule_id in future to reduce semantic ambiguity while retaining snapshot columns.
